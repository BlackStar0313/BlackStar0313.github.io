Jenkins使用pipelineGraphView尝试

写在前面​

最近在研究Devops流程，之前上班公司一直都是在用Jenkins，使用方式也都是freestyle项目，然后在配置中写shell脚本，在jenkins中手动执行或者轮询检查git提交，进行构建。

这种方式我觉得存在的问题：

  1. 配置在Jenkins中的shell脚本，没有版本控制，每建一个job就需要重新配置所有流水线，如果有人改动或者不小心把job 删了（有人干过这事儿，一个策划）。无法找回。构建逻辑一般很少改动，你想再想起来咋写可难了。
  2. 不够可视化，整个流水线怎么运作的有哪些过程，都需要阅读shell脚本（多数情况shell脚本里会套着执行项目里的打包脚本）.或者需要通过看jenkins打包log来看是到哪了。对熟悉整个流程的程序还好，如果是不了解的程序，包括非程序人员，几乎无法理解。
  3. 对于devops思想来说，需要自动化打包流水线可视化，鼓励所有人员参与进来。

所以我在自己家里的mac上通过docker跑了jenkins尝试下 pipleline graph view 这个插件。

pipleline graph view 插件.

jenkins docker 安装.

尝试使用pipelineGraphView​

一些重要步骤：

  1. 安装pipleline graph view 插件

    • Manage Jenkins -> Plugins -> Available Plugins-> 搜索pipleline graph view 安装

  2. 配置Credentials

    • Manage Credentials -> system -> Global credentials -> Add Credentials -> (看你是github还是ssh) SSH Username with private key ->
    • Scope: Global
    • ID: github-ssh
    • Username: git
    • Private Key: 粘贴私钥（无口令就留空）

  3. 创建一个pipeline项目

    • new item -> Pipeline -> 输入项目名称 -> 选择Pipeline script from SCM -> 选择Git -> 输入Git仓库地址 -> 选择分支 -> credentials 选择刚才创建的github-ssh -> Script Path 写仓库下新建的pipeline脚本文件名 -> 点击"保存"。

  4. 在自己git仓库下新建一个脚本（叫Jenkinsfile） 这里只是一个示例的pipeline脚本.展示了每2分钟轮询一次git仓库.

Jenkinsfile
pipeline {
	agent any
	tools { nodejs 'node18' }
	options { timestamps() }
	triggers { pollSCM('H/2 * * * *') } // 每2分钟轮询
	stages {
		stage('Build') {
			steps {
				sh 'npm ci'
			}
		},
		stage('Deploy') {
			steps {
				sh 'npm run build'
			}
		}
	}
	
	post {
    	always {
			dir('xxxx项目目录') {
				archiveArtifacts artifacts: 'build/**', allowEmptyArchive: true, fingerprint: true
			}
		}
	}
}
  1. git push脚本到仓库，然后在jenkins中构建。就可以看到最终的执行和pipleline graph view 的图。

one more thing​

给Piple 添加提交后自动进行构建​

通过github webhook 触发jenkins构建.

  1. jenkins配置中triggers 选择 GitHub hook trigger for GITScm polling （如果没插件先装github相关插件）
  2. github仓库配置webhook.
  • 仓库 -> settings -> webhooks -> add webhook
  • payload URL： http(s)://<你的jenkins域名或ip:端口>/github-webhook/
  • content-type：application/json 其他默认选项就行

ok ok 这样当github push 的时候，jenkins会自动进行构建。

总结​

整体上，使用Pipleline，脚本文件跟随项目在仓库里，流水线得点击去，看可视化，但是可是话比较清晰，每一步的log对应每个点位，而且如果报错，一眼就能定为到是构建的哪一步出错了，比freestyle项目需要看error然后自行定为大致范围直观很多。长得跟github Actions和gitlab 的CI/CD 很像。

缺点是需要去学习一下pipleine写法，而且可视化图入口比较深。
