ipa本地无法安装

写在前面​

在新发版本打包后，想要本地测试，但发现ipa通过xcode也好，通过Apple Configurator也好,还是 devicectl，都安装不上。非常头疼，整了一天多的时间找问题，头一次遇到。这里记录下寻找问题过程。

上下求索​

表现：

  • 打出的ipa, 通过xcode的 Devices and simulators 拖进去，提示安装失败
ERROR: Error Domain=com.apple.dt.CoreDeviceError Code=3002 "Failed to install the app on the device." UserInfo={NSUnderlyingError=0x6000036664f0 {Error Domain=IXRemoteErrorDomain Code=7 "Failed to stream the app asset from remote device" UserInfo={NSLocalizedDescription=Failed to stream the app asset from remote device, NSUnderlyingError=0x600003666040 {Error Domain=IXRemoteErrorDomain Code=7 "Failed to read the header before reading any data" UserInfo={NSLocalizedDescription=Failed to read the header before reading any data, NSUnderlyingError=0x600003666a60 {Error Domain=IXRemoteErrorDomain Code=7 "Received EOF when trying to read() 4 bytes"

  • 通过Apple Configurator 拖进去，提示安装失败

  • 通过devicectl 安装，提示安装失败
xcrun devicectl --timeout 900 device install app \  --device <UDID> ./<IPA_NAME>.ipa

  • 本地打包，然后通过xcode连设备使用dev证书直接run ，安装成功.
  • 将ipa 传到某个服务器上重新签名，通过设备上浏览器就行下载 itms-services://?action=download-manifest&url=... 安装成功。
  • 将ipa传到testflight 上，通过testflight 安装成功。
  • 将上一个版本的ipa ，通过上述几种方式尝试安装，都安装成功。

由于上个版本ipa都没问题，这个版本的ipa通过线连的方式有问题，首先先找下这两个版本添加和修改的内容有啥会影响打包的。

  • 因为app证书过期，重新出了adhoc证书。
  • 增加了一个新的小游戏在主包中。
  • 增加了一些常规运营活动的资源。

因为是跟装包有关的，首先怀疑的是新出的adhoc证书有问题。

将.ipa修改为.zip,然后解压，payload下是.app文件，进去查看内容，里面有个embedded.mobileprovision文件，这里会有所有出证书时该证书包含的所有测试设备udid.

但是发现自己的设备在其中，由于不是我自己出的证书，让出证书的同学重新帮忙出了个adhoc证书，重新打包了一下。但是结果还是安装失败,走不通。其实到最后找到具体问题之前证书的问题还没被排除。

接下来从报错入手，从上面各种安装方式报错的截图，每个提示的错误都不一样，挨个google找有没有类似情况，找了很久似乎相关的错误都不是很符合。走不通。

通过表象来找类似错误的方式都不行，只能硬着头皮找具体报错看具体是什么引起的了。

打开xcode的devices and simulators，选择自己的设备，打开对应设备的open Console,start后，将ipa拖到device里安装，报错，然后暂停console的log. 将全程log复制出来，查看。（这里log很多都看不懂，也是研究了很久，最后通过AI翻译，才找到关键点）

  1. 安装是走了streamingUnzip 远端流式解压

  1. 失败的位置 流式解压被中断了(IXRemoteErrorDomain 7，读 header 前就 EOF) 这个错误跟上面通过xcode安装和通过devicectl安装报错一样，可以互相印证.

如何印证真的就是因为这个解压的问题？

我把ipa先解压，然后将.app文件传到设备上,通过这种方式绕过streamingUnzip ，发现确实可以安装上。

	TMP=/tmp/ipa_to_app; rm -rf "$TMP"; mkdir -p "$TMP"
	unzip -q "./<IPA_NAME>.ipa" -d "$TMP"

	# 安装 .app
	xcrun devicectl --timeout 900 device install app \
	--device <UDID> "$TMP/Payload/<APP_NAME>.app" \
	--log-output /tmp/install_app.log

这样就可以确定，跟换证书无关，跟这个streamingUnzip 远端流式解压有关。 找到问题根源，然后就怀疑，是增加了资源，包体变大了？导致ipa通过本地安装装不上，搜了下苹果没有这个限制，而且其他项目ipa有超过1G的也没说装不上。经过一系列探索，原来是ipa内部文件数量的问题，超过了65535个文件.

unzip -l Your.ipa | wc -l        # 估算文件条目数

上图中，上面是安装不了的，下面是可以安装的上个版本包.

同样的工程环境，我把内部一些不影响打包的图片资源删了一下，重新打包，各种安装方式都成功了。至此问题找到。

原因​

什么是StreamingUnzip​

StreamingUnzip 是苹果官方提供的一种解压方式，它可以在解压过程中，将解压后的文件流式传输到设备上，而不是一次性将所有文件解压到本地。这样可以节省设备的空间，也可以提高解压速度。

安装 .ipa 时，把“压缩过的 IPA 数据”通过 usbmuxd 的 TCP 通道持续流到设备端，由设备上的 installd 子服务一边接收一边解压到临时目录，然后再校验、签名检查并提交安装。部分系统日志里会看到名字类似 com.apple.streaming_zip_conduit 或 IXSRemote*

流程：Mac(devicectl/Xcode) → usbmuxd(TCP over USB/Wi‑Fi) → 设备(installd/StreamingUnzipService)

超过65535个文件失败​

66480 已跨过 ZIP 的 65535 条目阈值，打包工具会启用 ZIP64 中央目录。StreamingUnzip 在部分系统/链路下对 ZIP64 更敏感/不稳，容易在长会话中断掉（表现为 EOF、IXSRemoteErrorDomain 7）.可以理解为StreamingUnzip是边解压边传输，zip64更大更不稳定，一但断了就安装失败.

为什么Testflight 可以安装​

Testflight 安装链路和本地安装不一样，使用的是商店通道，不走 streamingUnzip ，TestFlight/App Store 走的是 appstored/itunesstored + InstallCoordination 的“商店通道”。构建会被 Apple 服务器二次处理（重签名、切片 App Thinning、按机型/地区分发、可断点续传校验），设备端下载的是“已处理后的切片包”， 稳定性高且可续传，不依赖 usbmuxd。

为什么自己传到服务器通过网页可以安装​

安装链路与本地安装不一样，不走 streamingUnzip， 下载走的是可续传的http(s) 下载，即使网络短暂抖动也能继续；失败只会重试单个区段，不会导致整个会话作废。

  • Safari 打开 itms-services://?action=download-manifest&url=... → 交给系统的 itmsservicesd/appstored。
  • 设备先下载 manifest plist，读取 assets 里 software-package 的 IPA URL（可选 md5-size、md5s、filesize 等）。
  • 设备用系统下载器（nsurlsessiond）把 IPA 通过 HTTPS 拉到本机临时目录，支持断点续传（Range/ETag/resume data）、后台下载与校验。
  • 下载完成后，交给 installd 从“本地文件”进行安装（本地解压→校验→提交）。全程不经过 usbmuxd，也不走 StreamingUnzip 的“远端流式解压”。

为什么xcode run可安装​

xcode Run 走的是开发路径，不触发 StreamingUnzip ,直接copy .app目录并install ,或作增量/差异copy,分文件传输，传输时间短，颗粒小。

因为项目中只有个别的用了资源bundle,其他资源都是散着的，解决办法要么是大量资源改用bundle的形式打在一起，要么小图手动合成大图，要么就是删掉一些没用的资源。都可以减少最后打包后ipa文件数量。
