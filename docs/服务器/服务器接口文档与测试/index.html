服务器接口文档与测试

需求​

我的服务器使用Asp.net 开发的web服务器。接口越来越多，单独的curl测试记录不过来了，而且返回值和请求参数在代码里直接看不方便。需要有个地方专门记录接口相关说明。

之前上班时候用过一些工具，有一些是通过写代码的方式也接口文档，维护起来相当麻烦，后来干脆在语雀或者飞书用markdown写接口文档，好处是方便，想怎么写怎么写，只要客户端服务器两个同学能看懂就行。缺点是太随意了，没有规范，维护也是个问题。测试就直接用curl进行测试。反正接口调通了就完事儿了。

目前我独立开发一个游戏，如果手写的话，依然有维护的问题。而且独立开发，开发流程不像上班时候每一步都是按照最后上线标准做，很大概率先实现基础功能后面会通过重构代码补充一些细节。接口文档就会经常变动。

解决方案​

我需要一个可自动化维护的方式。并能方便的进行测试。

我选择了 服务器代码中添加接口，自动生成OpenAPI文档，然后在Apifox中导入，可视化。并可进行接口测试。

Apifox = Postman + Swagger + Mock + JMeter 的结合体

步骤​

  1. 在.net服务器代码程序启动时，添加扫描接口。
   builder.Services.AddOpenApi();
   var app = builder.Build();
   if (app.Environment.IsDevelopment())
   {
       app.MapOpenApi();
   }

   app.Run();
  1. 在接口代码后添加生成OpenAPI的说明 比如我这里一个login的接口
   app.MapGroup("/login")
       .MapGet(
           "/login",
           async (
               HttpContext ctx,
               string? DeviceId,
           ) =>
           {
                   // 登录逻辑
           }
       )
       .WithOpenApi(op =>    // OpenAPI接口参数
       {
           var param = op.Parameters?.FirstOrDefault(p =>
               string.Equals(p.Name, "DeviceId", StringComparison.OrdinalIgnoreCase)
           );
           if (param != null)
           {
               param.Required = true;
           }
           return op;
       })
       .Produces<ApiResponse<PlayerData>>(StatusCodes.Status200OK) // 返回具体数据类型
       .Produces<ApiResponse<object>>(StatusCodes.Status400BadRequest);  // error 返回具体数据类型

以上两步代码添加，保证自动导入的openapi 可以带入参数和返回值具体数据类型.

  1. 服务器 dotnet run 启动后，浏览器查看具体生成的OpenAPI json文件
http://localhost:5059/openapi/v1.json

  1. 导入Apifox中导入接口

每次更新，只需要一键导入，接口就全更新了，无需手动修改。

具体Apifox使用，在他里面点点就知道了，可以纯看文档，也可以不同url 进行接口测试，替代postman.
